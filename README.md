# UnscrambleGame
Обучающая игра-головоломка, разработанная на платформе Android с использованием современного подхода Jetpack Compose. Игроку предлагается угадать исходное слово, перемешанное случайным образом. Проект демонстрирует правильную архитектуру Android-приложений с использованием компонентов Jetpack, в частности ViewModel и StateFlow для управления состоянием и сохранения данных при изменении конфигурации устройства (поворот экрана).

## Технологии
- **Kotlin** — основной язык разработки
- **Jetpack Compose** — современный UI-фреймворк
- **ViewModel** — управление состоянием и жизненным циклом
- **StateFlow** — реактивное хранение состояния

## Функциональность
- Генерация случайного перемешанного слова из списка терминов Android-разработки
- Поле ввода ответа с валидацией
- Подсчёт очков (20 очков за правильный ответ)
- Отслеживание номера текущего вопроса (1 из 10)
- Отображение ошибки при неправильном ответе
- Возможность пропустить слово без начисления очков
- Диалоговое окно с результатами по окончании игры
- Кнопка «Играть снова» для перезапуска
- **Автоматическое сохранение состояния при повороте экрана** (благодаря ViewModel)
- Полное модульное тестирование ключевой логики

## Контрольные вопросы

### 1. Что такое ViewModel и зачем он нужен?
**ViewModel** — это компонент архитектуры Android Jetpack, предназначенный для хранения и управления данными, связанными с пользовательским интерфейсом.
ViewModel решает проблему потери данных при изменении конфигурации устройства (например, поворот экрана). Без ViewModel Activity уничтожается и создаётся заново, и все данные, хранящиеся в ней, теряются. ViewModel переживает эти изменения и сохраняет данные.
При использовании обычных переменных или remember { mutableStateOf() } внутри Composable-функций все данные сбрасываются, и счёт, текущее слово и прогресс игры возвращаются к начальным значениям.

### 2. В чём разница между MutableStateFlow и StateFlow?
- `MutableStateFlow` — изменяемый поток данных. Он имеет методы для изменения значения (value), что позволяет обновлять состояние.
- `StateFlow` — неизменяемый поток данных только для чтения. UI может подписываться на него и наблюдать за изменениями, но не может напрямую изменять состояние.
**Зачем создавать два поля: _uiState и uiState?**
Это реализация паттерна "инкапсуляция". Приватное поле `_uiState` (MutableStateFlow) используется внутри ViewModel для изменения состояния, а публичное поле `uiState` (StateFlow) доступно для UI только для чтения. Это защищает состояние от нежелательных изменений извне.
**Что такое “backing property pattern”?**
Это паттерн, при котором для одного свойства создаются два поля:
- Приватное (с подчеркиванием) — для хранения и изменения данных.
- Публичное (без подчеркивания) — для доступа к данным извне.
Это обеспечивает контролируемый доступ к состоянию и соблюдение принципа инкапсуляции.

### 3. Почему в ViewModel нельзя хранить Context или View?
ViewModel спроектирована так, чтобы переживать жизненный цикл Activity и Fragment. Если сохранить ссылку на Context (особенно на Activity context) или View в ViewModel, это приведет к тому, что ViewModel будет удерживать эти объекты даже после их уничтожения.
**Что может произойти, если сохранить ссылку на Activity в ViewModel?**
Произойдет **утечка памяти (memory leak)**. Activity после поворота экрана должна быть уничтожена сборщиком мусора, но ViewModel (которая живет дольше) продолжает хранить на нее ссылку, предотвращая очистку памяти. Это может привести к падению производительности и ошибкам OutOfMemoryError.

### 4. Чем StateFlow отличается от remember и rememberSaveable?
- **remember**: хранит состояние только в памяти Composition. При повороте экрана состояние сбрасывается. Используется для временного UI-состояния (например, развернут ли аккордеон).
- **rememberSaveable**: сохраняет состояние в Bundle. Выдерживает поворот экрана, но теряется при закрытии процесса (process death). Используется для состояния, которое должно пережить изменение конфигурации (например, введенный текст в поле поиска).
- **StateFlow**: компонент архитектуры, живущий в ViewModel. Выдерживает поворот экрана и используется для бизнес-логики и состояния, которое должно быть отделено от UI.
**Когда лучше использовать каждый подход?**
- `remember` — для простого локального состояния UI, которое не важно сохранять.
- `rememberSaveable` — для UI-состояния, которое должно пережить поворот, но не относится к бизнес-логике.
- `StateFlow` — для бизнес-логики и состояния, управляемого ViewModel.

###  5. Что такое data class в Kotlin и зачем он используется для UI State?
**Data class** — это специальный класс в Kotlin, предназначенный для хранения данных. Компилятор автоматически генерирует для него полезные методы.
**Какие методы автоматически генерируются для data class?**
- `equals()` / `hashCode()` — для сравнения объектов по содержимому, а не по ссылке.
- `toString()` — для удобного вывода содержимого класса.
- `copy()` — для создания копии объекта с изменением некоторых полей.
- `componentN()` — для деструктуризации.
**Зачем нужна функция copy()?**
`copy()` критически важна для работы с неизменяемым состоянием (immutability). Вместо изменения существующего объекта, мы создаем его копию с обновленными полями. Это позволяет легко отслеживать изменения состояния и избегать побочных эффектов.

### 6. Как работает collectAsState() в Compose?
`collectAsState()` — это функция, которая подписывается на поток StateFlow и преобразует его в State Compose.
**Почему UI автоматически обновляется при изменении StateFlow?**
Когда значение в StateFlow меняется, `collectAsState()` получает уведомление и обновляет свое внутреннее состояние. Compose отслеживает это состояние, и при его изменении автоматически запускает рекомпозицию (перерисовку) всех Composable-функций, которые используют это состояние.

### 7.  Зачем разделять код на пакеты data и ui?
Разделение на пакеты (data, ui, domain) — это реализация принципов чистой архитектуры.
**Какие принципы архитектуры это поддерживает?**
- **Разделение ответственности (Separation of Concerns)**: Каждый слой отвечает за свою задачу. `data` — за данные и модели, `ui` — за отображение и логику экрана.
- **Тестируемость**: Можно тестировать бизнес-логику (ViewModel) независимо от источников данных или UI-фреймворка.
- **Модульность и масштабируемость**: Код становится более организованным, его легче поддерживать, расширять и передавать другим разработчикам.

### 8. Что происходит с ViewModel, когда приложение полностью закрывается (process death)?
**Сохраняются ли данные в этом случае?**
Нет. ViewModel хранит данные только в памяти. Когда Android полностью убивает процесс приложения (из-за нехватки памяти или после долгого нахождения в фоне), все объекты ViewModel уничтожаются вместе с процессом, и данные теряются.
**Какие альтернативы есть для долговременного хранения?**
Для сохранения данных между сессиями приложения (при process death) используются другие механизмы:
- **SavedStateHandle** (в составе ViewModel): позволяет сохранять и восстанавливать небольшие объемы данных при process death, используя Bundle.
- **DataStore / SharedPreferences**: для хранения простых настроек и предпочтений.
- **Room Database**: для структурированных данных и сложных запросов.
- **WorkManager**: для отложенных фоновых задач.
